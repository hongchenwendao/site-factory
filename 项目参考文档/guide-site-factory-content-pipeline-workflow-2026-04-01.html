02

技术栈选型 — 为什么是这三件套

TECH STACK DECISION

前端 + 路由 + SSG

Next.js

React 框架，Vercel 亲儿子。SSG（静态生成）让每个页面都是预渲染 HTML——对 SEO 来说是最优解。ISR（增量静态再生）让你更新内容时不需要重新构建整个站。

SSG/ISR 动态路由 API Routes
内容管理

Sanity CMS

Headless CMS，内容和前端完全分离。Claude 通过 API 直接写入内容，不需要打开任何后台界面。免费额度：2 个 dataset + 100K API 请求/月，够用。

API 写入 实时预览 免费额度大
部署 + CDN

Vercel

Git push = 上线。全球 CDN，自动 HTTPS，预览部署。Next.js 的原生平台，零配置。免费额度：100GB 带宽/月/项目，个人和小流量站完全够。

一键部署 全球 CDN 预览环境
为什么不选其他方案
方案淘汰原因
WordPress学习成本高、插件依赖重、安全维护累。你自己说了"太累了"。
ShopifyB2B 不需要购物车/支付/库存。$39/月/站的成本在矩阵模式下叠加太快。SEO 博客能力弱。
Webflow / FramerCMS API 限制多，程序化 SEO 批量页面能力不足。Claude 对其代码生成支持不如 Next.js。
AI 一键建站10Web / Durable 等建站快但 SEO 控制弱，后续自动化接口有限，天花板低。

例外情况：如果某个客户站真的需要在线支付（B2C/DTC），可以在 Next.js 里接 Shopify Storefront API 或 Stripe，不需要换整套架构。

03

Phase 1 · 搭基座

BUILD THE FOUNDATION — 预计 2-3 天

这是一次性投入。搭好之后，后续每个新站都是 clone + 改 config。

项目初始化

Claude 执行 · 30 分钟
# Claude Code 执行以下命令
npx create-next-app@latest site-factory --typescript --tailwind --app --src-dir
cd site-factory
npm install @sanity/client @sanity/image-url next-sanity next-sitemap
npx sanity@latest init --project-id YOUR_ID --dataset production

你只需要做一件事：去 sanity.io 注册账号，拿到 project ID 和 API token。其余全是 Claude 的活。

Config 驱动架构

Claude 执行 · 2 小时

每个站的所有差异都集中在一份 config 文件里：

// site.config.ts — 这就是"换一个站"需要改的全部内容
export const siteConfig = {
  // 基础信息
  name: "Solar Panel Pro",
  domain: "solarpanelpro.com",
  locale: "en-US",

  // 品牌
  logo: "/logo.svg",
  colors: { primary: "#1E40AF", accent: "#F59E0B" },

  // 业务类型 — 决定页面结构
  type: "b2b-product",  // b2b-product | b2b-service | b2c-dtc

  // SEO
  defaultMeta: {
    titleTemplate: "%s | Solar Panel Pro",
    description: "Premium solar panels for commercial installations",
  },

  // 联系方式 / CTA
  cta: { text: "Request Quote", type: "form" }, // form | email | whatsapp

  // 内容管道配置
  content: {
    sanityDataset: "solar-prod",
    blogEnabled: true,
    productsEnabled: true,
    autoPublish: false,  // true = 免审直接发，false = 草稿等你审
  },

  // 导航
  nav: [
    { label: "Products", href: "/products" },
    { label: "Blog", href: "/blog" },
    { label: "About", href: "/about" },
    { label: "Contact", href: "/contact" },
  ],
}

模板页面搭建

Claude 执行 · 4-6 小时 · 你审核设计

Claude 生成以下页面模板，每个都自适应 config 中的 type 字段：

通用页面

  • 首页（Hero + 价值主张 + CTA）
  • 关于页（公司故事 + 团队）
  • 联系页（表单 + 地图）
  • 404 页

动态页面

  • 博客列表页 /blog
  • 博客详情页 /blog/[slug]
  • 产品列表页 /products
  • 产品详情页 /products/[slug]

你的决策点：审核首页设计稿，确认风格和布局。Claude 会出 2-3 个变体让你选。

SEO 基建

Claude 执行 · 1-2 小时
  • 自动 sitemap：next-sitemap 每次构建自动生成
  • 结构化数据：JSON-LD（Organization + Product + Article + BreadcrumbList + FAQ）
  • Meta 标签：每页自动生成 title / description / Open Graph / Twitter Card
  • robots.txt：自动生成
  • 内链系统：博客文章自动链接到相关产品页，产品页自动链接到相关博客
  • Answer-First 模板：博客文章结构强制 H2 问题 → 直接答案 → 展开内容（Brandon Leibowitz 的方法）

部署上线

Claude 执行 · 15 分钟
# 连接 Vercel
npx vercel link
npx vercel --prod
# 绑定域名（在 Vercel Dashboard 或 CLI）
npx vercel domains add solarpanelpro.com

完成。你的站现在在线了，全球 CDN，HTTPS 自动配好。

Phase 2 · 内容管道

CONTENT PIPELINE — 站建好了，开始灌内容

内容质量控制 — 不能全自动

铺量阶段也有底线。每篇文章 Claude 生成后，你需要 2-3 分钟扫一眼：

  • 事实准确性 — AI 编的数据会杀死你的 E-E-A-T 信号
  • 去 AI 味 — 用 /humanizer skill 过一遍
  • 底部漏斗检查 — 这篇文章的读者真的会买东西吗？
  • 内链检查 — 是否链到了 Money Page（产品页/服务页）？

当站点稳定后（3 个月+），可以把 autoPublish 设为 true，让信任度高的内容类型免审。

Phase 3 · 矩阵扩展

SCALE TO MULTIPLE SITES — 从 1 个站到 N 个站

基座搭好后，扩展新站的流程极其简单。本质上就是 复制 config → 改品牌信息 → 部署。以下是两种矩阵模式的操作手册。

模式 A · 同品牌子站

场景

chending.net 主站 + solar.chending.net 太阳能板 + tools.chending.net 工具站

操作

  • 同一个 Git 仓库，不同的 Vercel 项目
  • 共享设计系统，换 config 里的颜色/logo/导航
  • Sanity 同一个 project 下开不同 dataset
  • 主站做权重集中，子站做长尾覆盖
模式 B · 多品牌矩阵

场景

完全独立域名：solarpanelpro.com + rvparts.com + 客户站 xyz.com

操作

  • 同一个 Git 仓库（Monorepo），每个站是一个 app
  • 各自独立的域名/品牌/设计
  • Sanity 独立 dataset，数据完全隔离
  • 共享底层组件和 SEO 逻辑

新站上线 SOP（30 分钟 - 2 小时）

1

你：决定新站

品类、域名、目标市场、CTA 类型。一句话告诉 Claude。

2

Claude：生成 config

根据你的一句话，生成完整的 site.config.ts，包括品牌色、导航、Meta、CTA 配置。

3

Claude：创建 Sanity dataset

通过 CLI 创建新的 dataset，配置 schema（如果是新品类可能需要微调 schema）。

4

Claude：部署到 Vercel

vercel deploy --prod + 绑定域名。自动 HTTPS。

5

你：审核 → 上线

预览链接看一遍，没问题就是正式站了。然后启动内容管道开始灌博客。

Phase 4 · SEO 闭环

FEEDBACK LOOP — 从"灌内容"到"精准灌内容"

铺量 1-3 个月后，Google Search Console 会告诉你哪些内容有效。这时候从"盲打"切换到"数据驱动"。

SEO 闭环工作流（月度） Google Search Console │ ▼ ┌───────────────────┐ │ Claude 分析 GSC │ │ · 哪些词有展示 │ │ · 哪些词排 3-20 │──→ 低垂果实清单 │ · 哪些页 CTR 低 │ │ │ · 哪些词你没覆盖 │ │ └───────────────────┘ │ ▼ ┌──────────────────────┐ │ 三种行动 │ ├──────────────────────┤ │ A. 新文章 │ │ 发现未覆盖的高意图词 │ │ → 生成新博客 │ ├──────────────────────┤ │ B. 优化旧文章 │ │ 排名 3-20 的词 │ │ → 加内容/加内链/改标题 │ ├──────────────────────┤ │ C. 关闭无效页 │ │ 3 个月零流量的页面 │ │ → noindex 或删除 │ └──────────────────────┘ │ 重复每月一次

AI Overview 优化（Answer Engine Optimization）

2026 年的 SEO 不只是 Google 排名——你的内容需要被 AI（ChatGPT / Perplexity / Google AI Overview）引用。这就是 Brandon Leibowitz 的 Answer-First Method 的价值所在：

  • 每个 H2 写成问题形式
  • H2 下面第一句就是直接答案
  • 答案脱离上下文也能独立理解
  • FAQ 区域放在文章底部
  • 结构化数据标记 FAQPage schema

这套结构在博客模板里已经固化了，每篇文章自动遵循。

内链策略

每篇博客自动执行：

  • → 链接到 1-2 个产品页
  • → 链接到 2-3 篇相关博客
  • → 从产品页反向链接
  • → 形成 Pillar-Cluster 结构

Claude 在写文章时自动查询现有内容，生成相关内链。不需要手动。

日常运营节奏

WEEKLY ROUTINE — 你每周实际花多少时间

系统跑起来后，你的时间主要花在"看数据"和"做决策"上，不是"做事"上。以下是推荐的周节奏。

周一 · 内容日（1-2 小时）
0:00 - 0:15

Claude 跑关键词研究

分析竞品更新 + GSC 数据 → 输出本周选题列表

0:15 - 0:30

你审核选题

选 5-10 个题目，删掉不对的，可能加几个你知道的行业题目

0:30 - 1:00

Claude 批量生成文章

5-10 篇 Answer-First 格式博客，含全套 SEO 元素

1:00 - 1:30

你快速审核

每篇 2-3 分钟扫一遍，改明显错误，确认发布

1:30 - 1:45

Claude 发布

推送到 Sanity → 自动上线

周四 · 数据日（30 分钟）
0:00 - 0:10

Claude 拉 GSC + Analytics 数据

流量趋势、新关键词、排名变化

0:10 - 0:20

你看数据做决策

哪个站有起色？加注。哪个站没动静？考虑调整或关闭。

0:20 - 0:30

产品页更新（如需）

新产品数据 → Claude 处理 → 推送上线

月度：每月第一个周四做一次完整的 SEO 闭环分析（Phase 4），输出月报 + 下月内容策略调整。

你的时间分配：70% 看数据做决策，20% 审核内容，10% 给 Claude 指令。
不写代码，不改设计，不碰后台。

WHEN YOU NEED E-COMMERCE

当某个站需要在线卖货时，不需要换架构。在现有 Next.js 基座上叠加电商能力：

方案 1 · Shopify Storefront API

Next.js 做前端（你已经有了），Shopify 只当后端电商引擎。你的页面设计完全自定义，Shopify 只管支付、库存、物流。

适合：品牌 DTC 站，需要 Shopify 的电商基建。
费用：+$39/月（Shopify Basic）

方案 2 · Stripe Checkout

更轻量。直接在 Next.js 里集成 Stripe，产品少（< 50 SKU）时完全够用。不需要 Shopify 的重型基建。

适合：SKU 少的利基产品站。
费用：Stripe 交易手续费 2.9% + $0.30/笔，无月费。

决策规则：SKU < 50 且不需要物流集成 → Stripe。SKU > 50 或需要物流/库存管理 → Shopify Storefront API。

② Answer-First Method — H2 写成问题 → 第一句给直接答案 → 然后展开。让 AI Overview / ChatGPT / Perplexity 引用你。

按这个顺序走，不要跳步

Day 1

上午：装 Node.js + VS Code + Claude Code（30分钟）
下午：看 Programming with Mosh 的 Next.js 入门（1小时）

Day 2

注册账号 + 读 Steipete

上午：注册 Sanity + Vercel + GitHub（各 5 分钟）
下午：读 steipete 的 Just Talk To It（30分钟）
晚上：读你自己的笔记 guide-steipete-prompt-philosophy（20分钟）
目标：掌握"怎么和 Claude Code 协作"的哲学

Day 3-4

让 Claude 搭第一个站

读你的笔记 guide-olivier-croguy 的前 4 步 → 建品牌文件夹 → 写 CLAUDE.md

Day 7-8

跑通内容管道

读你的笔记中 Brandon Leibowitz 的 Profit Matrix + Answer-First

Day 9-10

学 Cody 的 Agent 思维

Day 11-12

矩阵验证

Clone 第一个站的 config → 改品牌信息 → 部署第二个站

目标：证明"一套代码 N 个站"可行

产品页 + 回顾

目标：完整的站（首页 + 产品页 + 博客 + SEO 基建）上线
